# Задача: оптимизация thebarcatering.cz по отчёту Lighthouse

Контекст: статический сайт (HTML/CSS/JS), хостинг Cloudflare Pages. Текущие баллы (mobile):
Performance 71 · Accessibility 95 · Best Practices 100 · SEO 92.
Цель — поднять Performance в зелёную зону (90+) и закрыть замечания по A11y и SEO,
**не меняя дизайн и вёрстку визуально**.

Метрики сейчас: LCP 5.6 s (плохо), FCP 3.5 s (плохо), Speed Index 3.8 s,
TBT 70 ms (ок), CLS 0 (отлично). Всё упирается в загрузку первого экрана.

Иди по приоритету сверху вниз. После каждого блока — проверяй, что сайт визуально не сломался.

---

## P1. Медиа героя — главный тормоз LCP (даёт самый большой прирост)

Общий вес страницы ~9.8 МБ, основной груз — фон/медиа первого экрана (это и есть LCP-элемент).

- Найди фоновое изображение/видео в hero-секции.
- Если это **картинка**: пересохрани в WebP или AVIF, отдавай через `<picture>` с
  responsive-размерами (`srcset` под мобильную/десктоп ширину). Цель — < 300 КБ на мобильном.
- Если это **видео**: добавь сжатый `poster` (LCP считается по нему), сделай видео
  `preload="none"` или подгружай после первого экрана; пережми битрейт.
- LCP-элементу добавь `fetchpriority="high"`, остальным фото — `loading="lazy"`.

## P2. Шрифты — 7 файлов, ~280 КБ, блокируют отрисовку

Сейчас грузятся и Google Fonts (через `fonts.googleapis.com/css2`, блокирует ~770 мс),
и 5 самостоятельных woff2 с домена (83.8 + 47.8 + 33.5 + 31.1 + 11.9 КБ).

- Всем `@font-face` добавь `font-display: swap` — текст появится сразу, не дожидаясь шрифта.
- `preload` только для 1–2 шрифтов, реально видимых на первом экране
  (заголовок «THE BAR.»): `<link rel="preload" as="font" type="font/woff2" crossorigin ...>`.
- Сократи число начертаний: грузи только те веса/стили, что реально используются.
- Google Fonts перестань грузить блокирующим `<link>`. Либо self-host этих шрифтов
  локально (как уже сделано с остальными), либо асинхронная загрузка
  (`media="print" onload="this.media='all'"`). Желательно — убрать зависимость от Google CDN целиком.

## P3. Render-blocking ресурсы (Est savings ~2 540 мс)

- `/style.css?v=43` блокирует рендер. Вынеси **critical CSS** первого экрана инлайном
  в `<head>`, остальной style.css подгружай асинхронно.
- Блокирующий запрос к Google Fonts CSS убирается вместе с P2.

## P4. Доставка изображений (Est savings ~143 КБ)

- Все `<img>` — в WebP/AVIF, с явными атрибутами `width` и `height`
  (предотвращает layout shift и убирает замечание из прошлого отчёта).
- Прогони картинки через сжатие (squoosh / sharp), убери оверсайз.

## P5. Forced reflow в JS (микро-оптимизация, не срочно)

Lighthouse указал конкретные места:
- `(index):116` — reflow 13 мс
- `(index):1379` — reflow 36 мс

В этих строках JS читает геометрию (`offsetWidth` / `offsetHeight` / `getBoundingClientRect`)
сразу после изменения DOM/стилей. Сгруппируй: сначала все чтения, потом все записи в DOM.
Где уместно — оберни в `requestAnimationFrame`.

## A11y. Доступность (сейчас 95)

- **Контраст в футере**: элементы `span` и `a` в `footer` не проходят по контрасту
  (приглушённый текст на тёмном фоне < 4.5:1). Сделай текст футера светлее
  или фон темнее до прохождения порога. Дизайн не ломать — подними яркость на пару ступеней.
- **Порядок заголовков**: heading-теги идут не по порядку (например, H1 → H3 минуя H2).
  Выстрой иерархию последовательно (H1 → H2 → H3), визуальный размер при этом
  оставь как есть через CSS-классы.

## SEO (сейчас 92)

- **Некликабельная для поисковика ссылка**: `a.bento-banner-cta` не краулится —
  скорее всего у неё нет настоящего `href` (навигация через JS/onclick или `href="#"`).
  Дай ей реальный `href` на целевую страницу/секцию.

## Баг вёрстки (вне Lighthouse)

- **Scroll-индикатор внизу первого экрана налазит на другие элементы.**
  Найди компонент (класс вроде `.scroll-indicator` / `.scroll-down`).
  Скорее всего `position: absolute` + `bottom` не учитывает высоту контента на мобильном.
  Добавь `pointer-events: none` (чтобы не перехватывал клики по кнопке выше),
  проверь `z-index` относительно кнопки REZERVOVAT AKCI, при нехватке места — спрячь
  индикатор на узких экранах через media-query.

---

## Порядок работы

1. Сделай отдельную ветку (например `perf/lighthouse-fixes`), не коммить в main напрямую.
2. Иди по P1 → P5, затем A11y → SEO → баг вёрстки.
3. После каждого блока — локальный прогон Lighthouse, сверяй LCP/FCP.
4. Визуально сравни до/после: дизайн и анимации не должны измениться.
